## Successes & Strengths

1. **Dynamic FilterRow architecture** — Replacing hardcoded `_filter_mfr`/`_filter_pkg` with a dynamic system that creates dropdowns based on `FILTERABLE_FIELDS` using `getattr()`. Adding a new filterable field requires only adding a tuple to the list — zero code changes elsewhere.

2. **Signal signature evolution** — Story 3.2 changed `search_requested` from `Signal(str)` to `Signal(str, str)`. All downstream connections updated systematically. Story 3.3 verified guided search compatibility without changes.

3. **`is_stale()` as a pure function in core/** — Stale detection logic is testable without GUI. The GUI just calls it and renders the result. Good core/GUI separation.

4. **`_StatusItem` subclass for custom sorting** — `QTableWidgetItem.__lt__` override with numeric sort keys (RED=0, AMBER=1, GREEN=2) is clean and maintainable for multi-level Qt table sorting.

5. **Column-index safety** — Using `VERIFY_COLUMNS.index("MPN Status")` instead of hardcoded column numbers. Story 3.4 added a "Freshness" column without breaking any existing column-dependent code.

6. **`blockSignals()` pattern propagated** — Story 3.1's FilterRow uses `blockSignals()` during dropdown population, applying the lesson from Epic 2's combo cascade bug.

## Key Insights & Learnings

1. **Dynamic over hardcoded** — FilterRow's `FILTERABLE_FIELDS` + `getattr()` pattern is the gold standard for extensible filter systems. Should be the default approach.
2. **QTableWidgetItem subclass for custom sorting** — `__lt__` override with numeric sort keys is the cleanest approach for Qt table sorting.
3. **Pure functions in core/ for time-based logic** — `is_stale()` pattern keeps testable business logic separate from rendering.
4. **Signal parameter growth is a code smell** — When a signal needs 3+ parameters, bundle them into a dataclass.
