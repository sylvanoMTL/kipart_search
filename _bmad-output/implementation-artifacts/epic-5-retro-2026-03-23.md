## Successes & Strengths

1. **Audit-first approach for existing code (Stories 5.1-5.2)** — Instead of reimplementing, these stories verified that existing code already met acceptance criteria. Result: 50 tests added validating existing behavior, zero production code changes. This was the right call.

2. **BackupManager in core/ with zero GUI deps** — The backup system (Story 5.5) is pure Python (json, csv, pathlib, dataclasses). Session-based semantics with `reset_session()` prevents duplicate backups. The undo log captures old values before writes — correct ordering.

3. **Schematic parser is production-grade** — `core/kicad_sch.py` handles escaped quotes, sub-sheet hierarchy, lock file detection, atomic writes, and binary I/O for line-ending preservation. Thoroughly tested (30 tests) and manually validated on real KiCad 9 projects.

4. **Dual-mode assignment (connected + standalone)** — Story 5.3's `_apply_assignment()` extraction handles both KiCad-connected (IPC write-back) and standalone (in-memory only) modes through one code path. Elegant and well-tested.

5. **Safety layering** — Multiple safety layers stack without conflicts: overwrite confirmation (5.4) → mismatch gate (5.4) → backup before write (5.5) → lock file detection (5.6/5.7) → atomic file writes (5.6) → undo log (5.5). Each layer is independently testable.

6. **Scope additions handled well** — Stories 5.6, 5.7, 5.8 were added mid-epic to address the limitation that IPC API is PCB-only. The schematic file parser enabled file-based write-back and dual-source scanning. Good decision to expand scope rather than ship incomplete.

## Challenges & Growth Areas

1. **Story 5.6 required 6 code review rounds** — The most review-intensive story in the entire project. Issues were mostly the same class of bug: regex patterns not handling escaped quotes in S-expression values. The fix was applied incrementally instead of systematically — each review found another instance of `[^"]+` that should have been `(?:[^"\\]|\\.)*`. Root cause: the initial implementation didn't establish a single pattern constant for escaped-quote-aware matching.

2. **Story 5.8 required 3 code review rounds** — Issues included: health percentage calculation not excluding desynced/sch-only components, `update_component_status()` overwriting sch-only visual state to green, datasheet field lookup not being case-insensitive.

3. **Pre-existing test failures accumulated** — By Story 5.6, there were 8 pre-existing GUI test failures from disabled IPC write-back (commit 4677e57), plus the original `test_central_widget_is_shrinkable_placeholder`. Test suite health degraded.

4. **Cross-story code bleed continued** — Story 5.7's "Push to KiCad" had dependencies on code from 5.5 (backup) and 5.6 (parser). The tight coupling between these stories made clean story boundaries difficult.

5. **Line ending corruption on Windows** — Story 5.6 code review #5 discovered that `Path.read_text()`/`write_text()` corrupts KiCad's Unix `\n` line endings on Windows. Required switching to binary I/O (`read_bytes()`/`os.write()`). This is a platform-specific gotcha that could have caused data corruption in production.

---

## Key Insights & Learnings

1. **Establish regex pattern constants early** — Story 5.6's repeated regex fixes could have been avoided by defining `_SEXPR_STRING_RE = r'(?:[^"\\]|\\.)*'` as a constant and using it everywhere from the start. When a pattern is used in >2 places, make it a constant.

2. **Binary I/O for file format preservation** — When modifying files that another tool owns (KiCad `.kicad_sch`), use binary I/O to preserve line endings, encoding, and BOM. `Path.read_text()` and `Path.write_text()` will silently modify line endings on Windows.

3. **Atomic file writes via tempfile + os.replace()** — Production-grade pattern for any file modification: write to temp file → `os.replace()` to destination. Crash-safe and avoids partial writes.

4. **Audit stories are valuable** — Stories 5.1 and 5.2 added 50 tests validating existing behavior without changing production code. This verified assumptions and caught edge cases (e.g., cache cleared on re-scan) that would have been bugs later.

5. **Scope additions mid-epic are OK when justified** — Adding Stories 5.6-5.8 was the right call. IPC API being PCB-only is a real limitation, and file-based write-back fills the gap. The key was: these stories had clear ACs and were implemented sequentially, not chaotically.

6. **Safety layers compose well** — Overwrite guards, mismatch gates, backups, lock detection, and atomic writes each serve a distinct purpose and don't interfere with each other. Layered safety is better than one monolithic guard.

## Significant Discovery: Schematic-Only Fields Need Sync Detection

During Epic 5, the team discovered that KiCad's IPC API (KiCad 9) provides PCB data only. Schematic fields (MPN, Manufacturer, etc.) are NOT available via IPC API — they can only be read from `.kicad_sch` files. This means:

- Components may have fields set in the schematic that the PCB scan doesn't see
- After F8 (Update PCB from Schematic), fields appear in PCB, but until then they're invisible to the IPC API
- The dual-source scan (Story 5.8) addresses this by reading both sources

**Impact on future epics:** Any feature that reads component fields must use the dual-source scan, not PCB-only. This is a fundamental architectural constraint.
