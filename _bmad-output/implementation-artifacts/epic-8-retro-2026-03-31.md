## Challenges & Growth Areas

1. **Update shim was overengineered** — The .bat script approach (process polling loop, exit code branching, self-delete in `%TEMP%`) was replaced entirely by `os.startfile()` during smoke testing. A single Windows API call handles installer launch, UAC elevation, and process independence. The complex approach was built from first principles without researching how existing Windows apps handle auto-update.

2. **10+ real-world bugs invisible to unit tests** — Fresh install path (`config.json` missing), Windows security (SmartScreen Zone.Identifier blocking), offline update check failure, cache edge cases, GitHub repo URL wrong (underscore), CMD window visible during shim, UAC elevation issues. All found during compiled binary testing, none caught by mocked unit tests.

3. **8 rounds of smoke testing to reach stability** — Distribution features have fundamentally different testing requirements than core logic. Mock-based unit tests gave false confidence about production readiness.

4. **PySide6 testing cheat sheet — 8 epics overdue** — Still not created, not referenced anywhere. Pre-existing GUI test failures noted in 5 of 8 stories (test_context_menus.py, test_assign_dialog.py, test_kicad_bridge.py).

5. **Worker thread cleanup pattern rediscovered 3 times** — Same `closeEvent` + wait-for-thread fix applied during code review in Stories 8.5, 8.6, and 8.7. Pattern not documented anywhere the dev agent can find it.

6. **No literature review for unfamiliar domain** — Epics 1-6 had strong research foundations via CLAUDE.md. Epic 8 entered new territory (Windows installers, auto-update, shell integration) without equivalent research. This directly caused the overengineering.

## Significant Discoveries

### 1. Update Shim Overengineering

The batch script shim approach (Story 8.7) — generate `.bat` in `%TEMP%` with tasklist polling loop, `/VERYSILENT` installer invocation, exit code checking, self-deletion — was **replaced entirely** by `os.startfile()` during end-to-end testing. Inno Setup's built-in `CloseApplications` and `/VERYSILENT` flags handle everything the batch script tried to manage manually.

**Root cause:** No literature review of how existing Windows applications handle auto-update before designing the solution. The dev agent reasoned from first principles instead of surveying proven patterns (VS Code/Squirrel.Windows, Electron updater, etc.).

**Impact:** ~359 lines of shim code reworked. 20 fix commits on `fix/update-mechanism` branch. 8 rounds of smoke testing to reach stability. Merged as PR #3.

### 2. Unit Tests Insufficient for Distribution Features

10+ production bugs were invisible to the unit test suite:

- `config.json` missing on fresh install (no prior state)
- SmartScreen Zone.Identifier blocking downloaded installer
- CMD window flashing during update
- UAC elevation path for silent install
- Offline update check crash
- Cache not saved when version already up to date
- GitHub repo URL wrong (underscore vs hyphen)

**Root cause:** Unit tests mock the filesystem, network, and Windows shell. Distribution features are defined by their interaction with these real systems.

---

## Key Insights & Learnings

1. **Literature review before story breakdown for unfamiliar domains** — Survey how existing tools solve the same problem before designing a solution. The .bat shim would never have been built if the team had first looked at how VS Code, Squirrel.Windows, or any Inno Setup-based updater works.

2. **Unit tests are necessary but not sufficient for OS-integration features** — Installer, update, shell, and security interactions require manual validation on real compiled artifacts. Mocked tests give false confidence.

3. **Knowledge discoverability > knowledge existence** — A cheat sheet nobody references is useless. Patterns must be in the dev agent's context path (project-context.md, CLAUDE.md) to be applied.

4. **Action items that aren't stories don't get done** — Proven across 8 epics. The PySide6 cheat sheet survived as a vague action item for 8 retros. Only stories with acceptance criteria get implemented.

5. **Simple solutions beat clever solutions** — `os.startfile()` replaced a 100+ line batch script. The simpler approach is more robust, more maintainable, and handles edge cases (UAC, file locks) that the complex approach had to solve manually.

## Team Agreements

- **Literature review before unfamiliar epics** — Epics entering new technical domains (OS integration, platform-specific, third-party tool) require documented research before story breakdown. Survey existing solutions, document proven approaches, store as planning artifact.
- **Distribution features require compiled artifact smoke testing** — Unit tests alone are insufficient. Install, update, shell, and security flows must be validated on real compiled binaries before marking the epic as done.
- **Knowledge discoverability** — Testing patterns, best practices, and architectural decisions must be referenced in `project-context.md` or `CLAUDE.md` to be discoverable by the dev agent. Documentation that isn't in the agent's context path doesn't exist.
- **Action items that survive 2+ retros become formal stories** — Enforced, no exceptions. (Carried from Epic 7, now proven necessary.)
- **Simple solutions over clever solutions** — Prefer proven platform APIs and existing tool capabilities over custom-built alternatives. Research first, build second.
