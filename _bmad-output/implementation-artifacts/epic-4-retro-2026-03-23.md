## Successes & Strengths

1. **Cache-aside pattern is clean and testable** — `SearchOrchestrator._search_sources()` and `verify_mpn()` both check cache before querying sources, with graceful fallback via `_cache_get()`/`_cache_put()` helpers that catch `sqlite3.OperationalError`. The cache is injected via constructor — easy to mock in tests.

2. **Atomic database replacement** — Story 4.2's download-to-temp-dir → backup-old → swap → restore-on-failure pattern prevents corruption during refresh. This is production-grade safety for a ~500MB file operation.

3. **PartResult serialization solved cleanly** — `part_result_to_dict()` and `part_result_from_dict()` handle nested dataclasses and Confidence enum serialization. The Enum `.value` conversion was caught during implementation (not in review).

4. **`is_local` property on DataSource** — Simple boolean property that lets the orchestrator and GUI distinguish local vs API sources without type-checking. Future API sources inherit `False` by default.

5. **Graceful degradation throughout** — Cache init failure doesn't crash the app, offline sources log warnings but don't block, corrupted database prompts re-download. Multiple layers of resilience.

6. **Code review fix rate improved** — 2/3 stories needed fixes (down from 100% in Epics 2-3). Story 4.1 passed review cleanly. The self-review checklist from Epic 2 may be having effect.

## Challenges & Growth Areas

1. **Confidence enum JSON serialization** — `dataclasses.asdict()` preserves Enum objects which aren't JSON-serializable. Caught during implementation but highlights that dataclass serialization isn't automatic when custom types are involved.

2. **Exception handling breadth** — Story 4.3 initially used `except Exception` for offline fallback. Code review narrowed this to `except (OSError, ConnectionError)` — catching too broadly can mask real bugs.

## Key Insights & Learnings

1. **Constructor injection for cache** — Passing `QueryCache` to `SearchOrchestrator` via constructor makes testing trivial (pass None or mock).
2. **WAL journal mode for concurrent access** — Essential for GUI + worker thread access to the same SQLite database. Set once at connection time.
3. **Atomic file operations for large databases** — temp-dir → backup → swap → restore pattern should be the standard for any file replacement operation.
4. **Narrow exception catches** — `except (OSError, ConnectionError)` is better than `except Exception` for network resilience — catches real offline scenarios without masking bugs.
5. **`is_local` as a property, not a method** — Simple boolean properties on ABCs are cleaner than `is_configured()` style method checks for source classification.
