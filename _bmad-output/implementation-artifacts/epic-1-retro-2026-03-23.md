5. **Single-file focus** — Most stories touched only `main_window.py` plus one panel file, keeping blast radius small and reviews fast.

6. **Anti-pattern documentation** — Each story spec included explicit "Anti-Patterns to Avoid" that prevented common mistakes proactively.

7. **Previous Story Intelligence** — Each story spec included learnings from the prior story, creating continuity and preventing re-discovery of solved problems.

## Challenges & Growth Areas

1. **PySide6 testing sharp edges** — Qt widget testing has non-obvious limitations discovered across 3/6 stories:
   - `isVisible` fails on unshown widgets — use `isVisibleTo(parent)` (Story 1.3)
   - `QMenu.exec` cannot be patched with `unittest.mock` — extract `_build_context_menu()` instead (Story 1.5)
   - `QAction` lacks `setAccessibleDescription()` — it's QWidget-only; use `setToolTip()` (Story 1.5)
   - `QTableWidgetItem` doesn't support `setAccessibleDescription()` — use `setToolTip()` (Story 1.5)

2. **QSettings test pollution** — Story 1.4 discovered that `QSettings` writes persist in the Windows registry across test runs. Required explicit `QSettings.clear()` in test teardown fixtures.

3. **Code review fix rate** — 4/6 stories (67%) needed post-review fixes. While catching issues is good, it suggests the dev pass could be tighter on: import conventions, objectName requirements, and cross-module access patterns.

4. **Deferred architectural decision** — Search dock position (Right vs Center, L1 from Story 1.1) was accepted but never formally tracked.

---

## Key Insights & Learnings

1. **Incremental story sequencing with shared patterns is highly effective** — Establishing foundations (objectNames, helpers, patterns) in early stories and reusing them eliminates rework.

2. **PySide6 testing gotchas should be documented once** — The same types of Qt testing surprises recurred across stories. A testing cheat sheet would prevent rediscovery.

3. **Anti-pattern documentation prevents mistakes proactively** — More cost-effective than catching issues in code review.

4. **Helper extraction is a testability breakthrough** — When a Qt method can't be mocked, extract the logic into a testable helper (e.g., `_build_context_menu()` returns a QMenu instead of calling `.exec()` directly).

5. **Story spec "Previous Story Intelligence" sections create accountability** — Each story builds on documented learnings, not tribal knowledge.
