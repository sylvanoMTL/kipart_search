## Successes & Strengths

1. **Core/GUI split for BOM export** — Stories 2.1-2.2 delivered a fully testable pure Python export engine (41 tests) before any GUI code was written. The engine has zero PySide6 dependencies.

2. **BoardComponent moved to core/models.py** — Fixed a long-standing boundary violation where a data model lived in `gui/kicad_bridge.py`. This enabled the export engine to import directly from core/ and unblocked clean architecture for future stories.

3. **Additive template pattern** — Story 2.2 added new templates by defining constants and extending `_group_components` with extra row fields. Existing PCBWay tests didn't break because writers use `.get(col.field, "")` — excellent API design.

4. **Health bar live update** — Story 2.3's `update_component_status()` avoids a full re-scan after each assignment. Sort-safe row lookup via UserRole data handles sorted tables correctly.

5. **DNP detection as BoardComponent property** — `is_dnp` property in `core/models.py` checks multiple field variants (dnp, do_not_populate, dnf). Business logic in core, not GUI.

6. **dataclasses.replace() for template overrides** — Elegant pattern for user-selected format/DNP without mutating preset constants.

## Challenges & Growth Areas

1. **Code review fix rate: 100%** — All 4 stories required post-review changes. Story 2.4 required two rounds. Common issues: dead code, naming consistency, signal interaction bugs, state sync.

2. **Signal cascade bug (Story 2.4)** — Changing template combo triggered format combo update, which re-triggered preview refresh. Required `blockSignals` fix. Qt signal chains need more careful upfront design.

3. **PySide6 visibility testing gotcha recurred** — Story 2.4 hit the same `isVisible()` vs `isHidden()` issue from Epic 1. The Epic 1 action item for a testing cheat sheet would have prevented this.

4. **Two code review rounds on Story 2.4** — Most complex GUI story to date. The export dialog had more moving parts (3 combos, preview table, file dialog, success message) and the interactions between them needed careful attention.

---

## Key Insights & Learnings

1. **Engine-first, then GUI** — Building the core BOM engine (Stories 2.1-2.2) before the dialog (2.4) meant the dialog was just a thin wrapper over well-tested core functions.

2. **Making internal functions public pays off** — Renaming `_group_components` → `group_components` let the preview table reuse the exact same logic as the actual export, ensuring preview matches output.

3. **Non-modal dialogs need reference retention** — `.show()` instead of `.exec()` means the dialog can be garbage-collected if not stored as an instance attribute.

4. **Signal cascades in combos** — When one combo's change triggers another combo's update, which triggers a refresh, use `blockSignals()` to prevent double execution.

5. **Code review fix rate trending upward** — 67% in Epic 1 → 100% in Epic 2. While the fixes are real issues, this suggests the first-pass dev quality could improve. Consider: self-review checklist before submitting.
